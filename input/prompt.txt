## Utility Suite
Create and utilize a file of testing utilities that is standard for all projects; name it `test_utils` and include all of the following:
```python
import pytest
from typing import Any
from abc import ABC, abstractmethod

class Mixin(ABC):
@abstractmethod
def condition(self, feature: str, *args: Any, **kwargs: Any) -> bool:
"""
The core method that tests a positive condition. It should
intercept a particular `feature` value and run its condition, but
leave non-intercepted features up to `super` to see if the next class
in the MRO can handle it.

Add arguments to allow injecting fixtures into the condition.
"""
raise NotImplementedError(f"No condition defined for the feature {feature}")

def validate(self, feature: str, *args: Any, **kwargs: Any) -> bool:
"""
The method that defined test methods should directly call. It will call the `condition` method and invert the result if `negative` is True.
"""

result: bool = False

expected_exceptions: tuple[type[BaseException], ...]

fail_for_feature: bool | type[BaseException] | tuple[type[BaseException], ...] = getattr(self, f"fail_{feature}", False)
negative_for_feature: bool = getattr(self, f"negative_{feature}", False)

if fail_for_feature is not False:
if fail_for_feature is True:
expected_exceptions = (BaseException,)
elif isinstance(fail_for_feature, tuple):
expected_exceptions = fail_for_feature
elif fail_for_feature is not False:
expected_exceptions = (fail_for_feature,)

with pytest.raises(expected_exceptions):
result = self.condition(feature, *args, **kwargs)
else:
result = self.condition(feature, *args, **kwargs)


if negative_for_feature:
return not result
else:
return result
```

## Feature Organization
Tests should be organized around "features" of the code being tested, each being implemented as a classes extending `Mixin`. You will need at least 3 classes per feature:
- `...MixinBase`: A base class that implements `condition`. Also define any fixtures needed for the condition here, but do not implement them. The class may also define `fail` with a non-False value, but classes that do so should have their names begin with `Fails`.
- `...Mixin`: A class that extends the base and sets `negative` to False.
- `No...Mixin`: A class that extends the base and sets `negative` to True.

All of these classes should be abstract.

The `...Mixin` and `No...Mixin` classes should be where direct `test_...` methods are defined. A `test_...` method should consist entirely of a call to `self.validate` with fixtures passed along.

Derive the `feature` value to intercept from the name of the base class. For example, if the base class is `IdentifiesEvenNumbersMixinBase`, the `feature` value to intercept should be `identifies_even_numbers`.

None of these defined classes should be test classes themselves, so they should either not begin with `Test` or define `__test__ = False`.

Additional classes may be defined as groups of mixins, and they should be empty abstract classes that extend the lowest-level mixin classes.

## Test Definition
The actual test classes should be defined separately from the mixin classes. They should be named based on the source class/method being tested and should extend one or more of the lowest-level mixin classes, but no other classes.

Subclasses should inherit all tests from their parent classes along with any new tests, with `negative` values being set for overwritten features, effectively ensuring that the override was implemented correctly. This is more clear and concise than manually putting `No...Mixin` classes in the inheritance list.

Implement all fixtures in these lowest-level test classes.

## General Testing Principles
- Conditions should be atomic - that is, designed to be as minimal as is meaningful. Keep them testing only a single aspect of the code's behavior.
- Features should also be relatively atomic, but in the sense of a conceptual group of behaviors rather than strictly a single aspect.
- Tests should be deterministic and not rely on any external state or side effects.
- Tests should be independent of each other, so that they can be run in any order without affecting the results.
- Tests should exclusively test the public interface and behavior of the code, not its internal implementation details.
- Add type hints whenever possible, and avoid wildcard types like `Any` or casts unless absolutely necessary.
- Prefer introducing `pytest-mock` if mocking is necessary, but avoid mocking if possible.
- Try to introduce at least one test file for every source file.
- Avoid non-class-based tests, `conftest.py` and `pytest.mark.parametrize` unless otherwise specified in the task prompt or necessary to meet coverage goals.
- Avoid assertion libraries - use plain `assert` statements.
- Keep code well-commented - explain non-obvious logic and decisions, and add docstrings to all test methods describing what they are testing and how they relate to the user's requirements.
